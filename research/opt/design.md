# RIFE / MetalFlow 최적화 설계 (O-트랙)

2026-07-24. 멀티에이전트 설계(비용분석 2 + 관점설계 5 + 3심판 채점)의 종합.
목표: ① M4에서 4K를 더 싸게 ② 저사양(M1/8GB)에서 실용 ③ GPU 경합 시 85ms 붕괴를 완만한 강등으로.

## 0. 실측 비용 구조 (설계의 근거)

조용한 M4, 60fps 소스, 120Hz 표시:

| 4K 전체화면 | work | 엔진 순수 비용 |
|---|---|---|
| RIFE | 30-31ms | ANE predict 10.9ms(360티어) + 경합 인플레 4-5ms, warp 2.5-4ms/t |
| MetalFlow | 31/36ms | flow 5.6ms + warp ~1ms = **~7ms뿐** |
| AppleFI | 24/39ms | VT 내장 ~10ms |

**핵심 발견 — 세 엔진이 24-31ms로 수렴하는 제1 원인은 해상도 준-무관 공유 지연 플로어(12-16ms)다:**
- SCK 배달 양자화: 소스 vsync 정렬로 8-16ms (구조적, 못 없앰)
- **틱 드레인 양자화: 평균 ~4.2ms (없앨 수 있음 — O1-1)**
- 720p에서 세 엔진 work 14-19ms인데 엔진 계산 2-7ms뿐 = 플로어의 직접 증거

4K 성장분(~8-12ms)은 풀프레임 트래픽: 소스 주기당 33MB 터치 ~11회 ≈ 360MB/16.7ms ≈ 22GB/s.
M4(120GB/s)에선 20%라 부차적, **M1(68GB/s)에선 35% — 저사양의 진짜 병목.**
GPU 경합 붕괴(85ms)는 이 플로어의 큐 대기 성분이 팽창하는 것 — 커널 최적화가 아니라
공유 경로·강등이 대책이다.

신규 발견: MetalFlow가 stableA(=직전 쌍의 stableB)의 루마 피라미드+zero-mean을 매 쌍
재계산 (MetalFlowEngine.swift:167-188). 핑퐁 재사용으로 ~0.5ms 공짜.

## Phase O1 — 공유 플로어 제거 (전 엔진 공통 이득, 페이싱 무접촉)

### O1-1. 캡처 콜백 구동 인제스트 (틱 양자화 제거) — 최대 단일 지렛대
- **현재**: SCK 콜백이 pendingSlots에 쌓고, 렌더 틱(120Hz)이 present 후에 drain → 인코딩 시작이
  평균 4.2ms(0~8.3ms) 늦음. work 지표와 e2e에 그대로 반영.
- **변경**: SCK 콜백 도착 시 `CFRunLoopPerformBlock`(async)으로 렌더 스레드 런루프에 인제스트
  블록을 직접 태움. **같은 렌더 스레드에서 실행**되므로 timeline/snap/slotStamps의 무락 전제
  전부 보존 (RenderDriver performSync 74-85행 구조가 async 변형을 지지). present 선택은 여전히
  틱이 함 → σ 불변.
- **기대**: 전 엔진 work −4ms급 + paceWorkP90 하락이 requiredExtra 하한(1060-1066)을 낮춰
  표시 지연까지 회수하는 2차 이득.
- **리스크/완화**: 버스트 시 인제스트 인코딩 CPU가 틱 직전에 끼는 지터 — drain depth 계측 +
  `MACFG_CBINGEST=0` 폴백 토글로 관리.
- **검증**: tsrc 균일/버스트에서 work·e2e·σ 전후 비교. σ=0.00 유지 필수.

### O1-2. MetalFlow A쪽 피라미드 핑퐁 재사용 (S)
- stableA는 직전 쌍의 stableB와 동일 텍스처 — downLuma/downHalf/zeroMean 재계산을 연속성
  체크(tsA==직전 tsB)로 스킵, 루마/zm 버퍼 핑퐁 스왑. ~0.5ms + 4K 읽기 1회분 절감.
- 검증: InterpBench 픽셀 동일성(연속 쌍) + tsrc σ.

### O1-3. 3-큐 분리 (cb2 event-wait head-of-line 해소) + uiMask 핑퐁
- 알려진 백로그의 실행 사양: cb1 stable blit을 전용 copy 큐로 분리해 cb2의
  encodeWaitForEvent(10-16ms 정지)가 다음 프레임 블릿을 막지 않게. **함정 선해결**:
  UIStaticDetector.maskTex 단일 텍스처가 큐 분리 시 레이스 (UIStaticDetector.swift:79) →
  마스크 핑퐁 + 세대 전달 (1프레임 지연은 α=0.04 EMA라 체감 불가). cb2들 간은 workQueue
  직렬 유지 (RIFE temporal prev 전제 보존).
- 기대: 정상시 중첩 복원, **경합 붕괴(85ms)의 구조적 증폭기 차단** — 저사양 목표 직결.
- 검증: MTL_DEBUG_LAYER 하자드 검사 + tsrc 경합 시뮬(동시 GPU 부하)에서 work 분포.

## Phase O2 — 저사양 거버너 트랙 (LoadGovernor 인프라 + 소비자들)

### O2-1. LoadGovernor (공유 부하 거버너) — 인프라, 최우선
- 입력 신호는 전부 기존 계측 재사용(비용 0): paceWorkP90, tickHz, diagTickGaps, paceMissCount,
  glass σ. AIMD + 히스테리시스(adaptPacing/RIFE maybeAdapt에서 검증된 패턴).
- 레벨 L0(전부 정상)→L1→L2→L3. 각 레벨은 인코딩 파라미터만 조작 — σ·지연·색 불변.
- **이중 강등 조정 규칙 필수**: RIFE 자체 사다리(maybeAdapt)와 같은 다이얼을 두 컨트롤러가
  만지면 발진 — 거버너는 상한(캡)만 내리고, 엔진 자율 사다리는 그 캡 안에서 동작.
- 수동 오버라이드 + `MACFG_GOV_FORCE`(강제 레벨, A/B용).
- 검증: 유닛(신호→레벨 전이표) + tsrc + 인위 GPU 부하에서 강등/복귀 이력 로그.

### O2-2. MetalFlow flowBase 런타임 사다리 (거버너 소비)
- 960→800→640→480. mfMatch L0 gather가 flow 5.6ms의 ~70%, base² 비례 —
  M1 최대 단일 손잡이 (M4 −2.4ms, M1 환산 −7~10ms 추정).
- ensureResources 키에 base 추가하되 **outputPool/statsBuffers 보존** (표시 대기 텍스처
  무효화 = leaseDrop 방지). prior 리셋은 전환 1쌍 한정.
- 화질 게이트: InterpBench SSIM 허용선 −0.01. L1 이하에서만 발동, 즉시 복귀.

### O2-3. 업스케일 체인 자동 강등
- MetalFX가 present 틱마다 실행 (RenderSurface 160-176) — 720p→4K M1 뷰어 시나리오 최대 회수.
- L1: MetalFX 끔, **ANE 2x 유지** (1.68ms GPU-free 실측) → 체감 최소. scaleStatus 자동 반영.

### O2-4. 보간 배율 자동 강등 + L3 보간 바이패스 — 최후 방어
- L2: 배율 상한 강등(4x→2x, 균등 스트라이드 = 소스 그리드 유지 → σ 무회귀).
  기존 배압 솎아내기(1489-1491) 패턴 재사용.
- L3: 보간 바이패스(원본 패스스루). **엔진 prepare는 유지** → 복귀 즉시(전환 공백 0,
  configurePairEngine 수백 ms/사다리 ~1s 재준비와 대비). "성능 보호 중" 표기.
- 이게 85ms 붕괴의 올바른 종착 — 깨진 보간보다 매끈한 원본.

### O2-5. 저사양 스마트 기본값 (기기 프로파일 시딩)
- 칩/메모리로 보수 시작값 시딩(정착값은 거버너 실측이 결정) — RIFE의 "보수 시작 후 실측
  승격" 철학(216-227)의 앱 전역 확장. 초기 miss 폭주로 +4슬롯 헛램프(1036 주석)도 차단.

### O2-6. 표시 목표 해상도 워프
- 표시 < 소스(M1 노트북 패널 + 4K 소스)면 워프를 표시 해상도로 — 두 워프 커널 모두 정규화
  UV + 출력 크기 기준 flowScale이라 **셰이더 무수정** (RIFEEngine 639, mfWarp 701).
  출력 풀 531→236MB (M1 8GB에 구조적 이득). 4K 1:1에는 미발동(무해).
- **게이트 필수**: S(프레즌트 미니파이)/I(타깃 워프)의 선명도 교차 플리커 — 그래디언트 지표
  + SSIM 0.99로 게이트, 실패 시 미적용.
- 방금 넣은 "4K Neural 고정 경고" UI는 이 트랙 완성 시 거버너 실측 기반 표시로 대체.

## Phase O3 — RIFE 모델 트랙 (predict 자체를 치는 정공법)

### O3-1. v3 재수출 — 블록 간 full-res 왕복 제거 (피라미드 체이닝)
- v2 forward가 블록마다 풀해상도 업샘플→재다운 왕복 (연구 코드 69, 105-111행 확인).
  conv-only 플로어 6.9ms(bench_m1est.py) vs v2 360 p50 10.9ms — 배관이 ~4ms.
- 피라미드 체이닝으로 왕복 제거 → 540티어 −5~7ms면 **30fps 2앵커가 540에서 성립**
  (사다리 상단 재편). v1→v2 −47% 전례가 방법론 신빙성.
- 게이트 3종(SSIM/워스트트윈/ANE 상주) + 벽시계 A/B 판정 + 모델 파일 롤백 무비용.

### O3-2. 사다리 무봉합 전환 + 웜업 + 플래핑 히스테리시스 (백로그 #45 흡수)
- 전환 창 ~1s 보간 공백(kickSwitch 342-375, 389 nil 반환) 제거: 백그라운드 프리페치 후
  원자 스왑 (encodePair의 원자 스냅샷 구조가 이미 지지).
- 4K predictMed 14.7-16.0 vs 강등 문턱 15.03 걸침 → 플래핑 개연 — 히스테리시스 폭 추가.
  **사전 진단으로 플래핑 실재부터 로그 확정** 후 착수.
- ⚠️ 스왑 시 인플라이트 쌍의 `[weak model]`이 nil화되어 50/50 강등 가능 — 구모델 강한
  참조를 드레인 완료까지 유지할 것 (심판 지적).

## 기각/보류
- MetalFlow 적응 flowBase의 중복 판본 2개 — O2-2로 일원화 (엔진 내부 자체 AIMD는 거버너와
  같은 다이얼 이중 조작 = 발진 구조라 기각. cb2 GPU 실측 신호 아이디어만 흡수).
- 엔진 내부 미시 최적화(1ms 미만 항목들) — 심판 규칙상 제외.

## 기대치
- **M4 4K**: work 30ms → O1 후 ~24-26ms (틱 양자화 −4, 핑퐁 −0.5, 큐 분리는 경합시 이득).
  잔여 플로어는 SCK 배달 양자화(소스 vsync 특성, 구조적).
- **M4 4K RIFE 60fps 2x**: O3-1 성공 시 predict 10.9→6-7ms — 17ms 예산에 여유.
- **M1**: O2 트랙으로 "느리지만 안 깨짐" — flowBase 640 + 표시해상도 워프 + 업스케일 강등이
  대역폭 35%→~20% 감축. 최후엔 L3 바이패스로 원본 유지.
- **경합 붕괴**: 85ms 방치 → 거버너가 L1~L3로 계단 강등, 복귀 즉시.

## 리스크/되돌림
- 모든 O1/O2 항목에 env 폴백 토글 (`MACFG_CBINGEST`, `MACFG_GOV_FORCE` 등) — 문제 시 기존
  경로로 즉시 복귀.
- O3-1은 모델 파일 교체라 롤백 무비용. 실패 게이트(SSIM/워스트트윈) 미달 시 v2 유지.
- 각 단계 커밋 전 tsrc 무인 검증: σ=0.00(균일), srcInt 락, work/e2e 전후 비교.

## 구현 순서 제안
O1-1 → O1-2 (반나절, 즉시 이득) → O2-1+O2-4 (거버너 골격+최후방어 먼저) → O2-2/3/5 →
O1-3 (큐 분리, 신중히) → O2-6 → O3-1 (모델 재수출, 별도 세션) → O3-2.
